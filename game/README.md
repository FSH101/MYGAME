# Wasteland: Survival

Браузерная мультиплеерная 3D-выживалка под мобильные устройства. Клиент и сервер написаны на TypeScript, рендеринг выполняется Babylon.js, сеть реализована через Socket.IO с авторитарной симуляцией на Node.js.

## Выбор технологий
- **Babylon.js** — современный WebGL/ WebGPU движок с готовыми мобильными оптимизациями (LOD, инстансинг, PBR), удобным API и малым объёмом ядра. Он быстрее выводит сложные сцены на телефонах, чем "голый" Three.js, благодаря встроенному сценографу и поддержке Touch.
- **Socket.IO** — проверенная обёртка над WebSocket с автопереподключением, fallbacks и удобной типизацией. Ускоряет разработку авторитарной модели, а также даёт бэкап-транспорт для мобильных сетей.
- **Fastify** — лёгкий HTTP-сервер с хорошей производительностью и встроенными хуками для логирования. Компактно обслуживает статический клиент и API.
- **cannon-es** и собственная ECS — позволяют контролировать бюджет физики и памяти, не затягивая тяжёлые зависимости.

## Требования окружения
- Node.js ≥ 20
- npm ≥ 9
- Docker (для контейнеризации)

## Установка и запуск
```bash
cd game
npm install
```

### Разработка
- `npm run dev:client` — Vite dev-сервер клиента (порт 5173)
- `npm run dev:server` — авторитарный сервер (порт 8080)

### Сборка
- `npm run build:client` — продакшн-бандл клиента с code-splitting и минификацией
- `npm run build:server` — компиляция сервера в `server/dist`
- `npm run build` — последовательная сборка клиента и сервера

### Docker
- `npm run docker:build` — сборка образа `wasteland-survival`
- `npm run docker:run` — запуск контейнера (порт 8080)

### Деплой на Render
1. Запушьте репозиторий и создайте новый **Web Service** на [Render](https://render.com/).
2. На шаге конфигурации укажите:
   - **Root Directory:** `game`
   - **Environment:** `Node`
   - **Build Command:** `npm install && npm run build`
   - **Start Command:** `npm run start`
   - **Instance Type:** выберите `Starter` или выше (минимум 512 MB RAM).
3. В разделе переменных окружения добавьте `NODE_VERSION=20` и при необходимости `NODE_ENV=production`.
4. Включите WebSocket-поддержку (Render делает это автоматически для Web Service) и установите **Health Check Path** в `/health`.
5. После первого деплоя убедитесь, что в логах видно успешную сборку клиента и сервера, затем откройте сгенерированный домен — клиент будет обслуживаться тем же сервисом, а WebSocket соединится по тому же хосту.

> ⚙️ Альтернатива: импортируйте `render.yaml` из корня репозитория — Render автоматически создаст сервис с теми же настройками.

### Нагрузочное тестирование
- `npm run test:load` — спавнит до 16 ботов (поведение прописано в `server/src/sim/bots.ts`). Параметр количества можно передать: `npm run test:load -- 12`.

### Модель игрока
- Рабочий манекен создаётся процедурно скриптом `npm run assets:generate`, который записывает `client/public/assets/models/player.glb`. Скрипт запускается автоматически перед `npm run dev:client` и `npm run build:client`, так что бинарник не хранится в git.
- GLB содержит три клипа (`idle`, `run`, `attack`) и простую геометрию (тело + шлем + ранец). При необходимости можно изменить скрипт генерации или заменить итоговый `player.glb` на собственный файл — загрузчик возьмёт любой корректный glTF/GLB по тому же пути.
- Каталог `client/public/assets/models/` хранится пустым (в репозитории лежит только `README.txt`) — добавляйте туда свои `.glb/.gltf` или `.fbx` вместе с текстурами только локально. Загрузчик автоматически сопоставит карты с суффиксами `_albedo`, `_normal`, `_rough`, `_metal`, `_ao` и выполнит конвертацию FBX в рантайме.
- Если `player.glb` отсутствует (скрипт не запускали) или повреждён, игра подставит капсулу-заглушку и выведет предупреждение в логах загрузочного экрана — после повторного запуска с генерацией всё восстановится.
- После замены модели перезапустите dev-сервер или запустите `npm run build:client`, чтобы обновить ассеты.

## Управление и UX
- **Левый “плавающий” стик.** Первый палец в левой половине (для левшей — в правой) спавнит базу стика в точке касания. Радиус и мёртвая зона учитывают DPR устройства. Двойной тап вперёд включает/выключает авто-бег, длинные радиусы включают спринт, малые — шаг.
- **Правая зона обзора.** Свайпы управляют поворотом камеры по экспоненциальной кривой чувствительности (γ ≈ 0.85). Вертикаль клампится, есть инверсия по настройке. Минимальные сдвиги (<2 px, <120 мс) игнорируются.
- **Кнопки действий.** Стилизация в едином стеклянном стиле (радиус 24 px, подсветка удержания), расширенная hit-area +12 px. Раскладка по умолчанию (портрет):
  - «Удар» — bottom: **3%**, right: **3%**, удержание = авто-атака, вибро 12 мс.
  - «Прыжок» — bottom: **16%**, right: **3%**, одиночный тап.
  - «Присесть/Лечь» — bottom: **16%**, right: **18%**, короткий тап = присесть, долгий (≥350 мс) = лечь/встать.
  - «Взаим.» — top: **45%**, right: **3%**, контекстное взаимодействие/сбор.
  - «Инв.» — top: **8%**, left: **3%**, переключатель мини-инвентаря (торчит как toggle-кнопка).
- **Левша/портрет/альбом.** Верстка перестраивает якоря кнопок по процентажам, зеркалирование активируется в настройках.
- **Клавиатура (desktop dev).** WASD — движение, Space — прыжок, F — удар, E — взаимодействие, I — инвентарь (для отладки).

UI адаптируется под DPR мобильных экранов, использует HTML/CSS поверх WebGL-канваса. Для отладки можно подсветить активные кнопки через классы `.hud-button__control.active` и `.hud-button__control.toggled`.

### Настройки управления
Файл `client/src/input/settings.ts` хранит профиль в `localStorage` (`wasteland-touch-settings`). Доступные параметры:

| Параметр | Диапазон | Описание |
| --- | --- | --- |
| `MoveDeadzonePx` | 8–30 | Радиус мёртвой зоны стика в физических пикселях |
| `MoveSprintThreshold` | 0.6–0.9 | Нормализованная длина вектора, при которой включается спринт |
| `MoveCurve` | linear / expo | Кривая плавности движения (expo сглаживает микро-движения) |
| `LookSensitivity` | 0.2–2.0 | Базовая чувствительность камеры |
| `LookGamma` | 0.6–1.2 | Экспонента для точной наводки |
| `InvertY` | bool | Инверсия вертикали обзора |
| `GyroAim` | bool | Зарезервировано под гироскоп (off по умолчанию) |
| `LeftHanded` | bool | Зеркальное расположение стика/кнопок |

Настройки можно править из devtools, модуль реагирует на изменения в рантайме (subscribe на стор).

### Экран загрузки и логирование
- При старте игры отображается полноэкранный загрузчик с прогресс-баром, журналом этапов (`Регистрация кеша`, `Инициализация рендера`, `Создание сцены`, `Загрузка ассетов`, `Инициализация сети`, `Создание игрока`, `Запуск игрового цикла`) и ошибками.
- Логгер (`client/src/core/Logger.ts`) транслирует `info/warn/error` в этот экран и в консоль. Кнопка «Скопировать лог» копирует текущую историю (до 400 строк) в буфер обмена.
- При ошибке этапа строка подсвечивается красным, ошибка разворачивается в блок `Ошибки`. Кнопка «Повторить» очищает SW-кеши (`caches.*`, `navigator.serviceWorker.unregister()`) и перезагружает страницу с параметром `?retry=...`.
- Для теста отказа можно временно удалить/переименовать `client/public/assets/models/player.glb` и перезагрузить страницу — этап «Загрузка ассетов» подсветится красным, в ошибках появится 404 и активная кнопка «Повторить».

## Игровые системы М1
- Авторитарный сервер с тикрейтом 20 Гц, снапшоты 10 Гц. Клиент интерполирует позиции игроков и сущностей.
- Карта 500×500 м, процедурный рельеф дюн, расставленные скалы и башни.
- Параметры игрока: HP, Hunger, Thirst, Stamina, температура окружающей среды. Температура влияет на расход жажды и стамины.
- Сбор ресурсов (камень, дерево, металл, вода, волокно), респавн нод каждые 120 сек.
- Крафт сервер-сайд, рецепты из таблицы ниже, инвентарь 4×5 слотов, стаки по типам.
- Моб «пустынная крыса» патрулирует и агрится в радиусе 18 м, наносит урон при контакте.
- Смерть сбрасывает часть слотов, респавн через 10 сек на случайной точке пустоши.
- День/ночь — синусоидальная кривая освещения, температура днём выше.
- PWA: manifest + service worker (кеш статических ассетов, офлайн-страница).
- Логирование FPS и ключевых метрик в консоли (devtools), сервер логирует тик/температуру каждые 5 сек.

```json
{
```

Фактическая реализация использует типизированные интерфейсы из `shared/types.ts`, снапшоты интерполируются клиентом. Все проверки крафта, урона и сборов происходят на сервере.

## PWA и оптимизации
- Service Worker кеширует shell и офлайн-страницу, лениво загружаются игровые чанки.
- Vite сборка с `splitVendorChunkPlugin`, ручное разбиение на чанки `babylon` и `net`.
- Lazy-load сцены (`import("./scenes/gameScene")`) и отложенная регистрация SW.
- Бюджет draw calls ≤ 200 достигается батчингом примитивов и упрощёнными материалами.

## Перспектива М2
- Подключение Prisma + SQLite для хранения инвентаря и построек.
- Переключатель настроек графики, дополнительные погодные зоны и пост-эффекты.
- Расширенная физика с помощью cannon-es: slope limit, step offset.
