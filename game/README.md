# Wasteland: Survival

Браузерная мультиплеерная 3D-выживалка под мобильные устройства. Клиент и сервер написаны на TypeScript, рендеринг выполняется Babylon.js, сеть реализована через Socket.IO с авторитарной симуляцией на Node.js.

## Выбор технологий
- **Babylon.js** — современный WebGL/ WebGPU движок с готовыми мобильными оптимизациями (LOD, инстансинг, PBR), удобным API и малым объёмом ядра. Он быстрее выводит сложные сцены на телефонах, чем "голый" Three.js, благодаря встроенному сценографу и поддержке Touch.
- **Socket.IO** — проверенная обёртка над WebSocket с автопереподключением, fallbacks и удобной типизацией. Ускоряет разработку авторитарной модели, а также даёт бэкап-транспорт для мобильных сетей.
- **Fastify** — лёгкий HTTP-сервер с хорошей производительностью и встроенными хуками для логирования. Компактно обслуживает статический клиент и API.
- **cannon-es** и собственная ECS — позволяют контролировать бюджет физики и памяти, не затягивая тяжёлые зависимости.

## Требования окружения
- Node.js ≥ 20
- npm ≥ 9
- Docker (для контейнеризации)

## Установка и запуск
```bash
cd game
npm install
```

### Разработка
- `npm run dev:client` — Vite dev-сервер клиента (порт 5173)
- `npm run dev:server` — авторитарный сервер (порт 8080)

### Сборка
- `npm run build:client` — продакшн-бандл клиента с code-splitting и минификацией
- `npm run build:server` — компиляция сервера в `server/dist`
- `npm run build` — последовательная сборка клиента и сервера

### Docker
- `npm run docker:build` — сборка образа `wasteland-survival`
- `npm run docker:run` — запуск контейнера (порт 8080)

### Деплой на Render
1. Запушьте репозиторий и создайте новый **Web Service** на [Render](https://render.com/).
2. На шаге конфигурации укажите:
   - **Root Directory:** `game`
   - **Environment:** `Node`
   - **Build Command:** `npm install && npm run build`
   - **Start Command:** `npm run start`
   - **Instance Type:** выберите `Starter` или выше (минимум 512 MB RAM).
3. В разделе переменных окружения добавьте `NODE_VERSION=20` и при необходимости `NODE_ENV=production`.
4. Включите WebSocket-поддержку (Render делает это автоматически для Web Service) и установите **Health Check Path** в `/health`.
5. После первого деплоя убедитесь, что в логах видно успешную сборку клиента и сервера, затем откройте сгенерированный домен — клиент будет обслуживаться тем же сервисом, а WebSocket соединится по тому же хосту.

> ⚙️ Альтернатива: импортируйте `render.yaml` из корня репозитория — Render автоматически создаст сервис с теми же настройками.

### Нагрузочное тестирование
- `npm run test:load` — спавнит до 16 ботов (поведение прописано в `server/src/sim/bots.ts`). Параметр количества можно передать: `npm run test:load -- 12`.

## Структура репозитория
```
/game
  package.json               # корневой monorepo + скрипты
  /client                    # Vite + Babylon клиент
    index.html
    vite.config.ts
    src/
      ecs/                   # ECS-ядро клиента
      net/                   # сетевые адаптеры
      systems/               # игровые системы (рендер, инпут, день/ночь)
      scenes/                # конфигурация Babylon-сцены
      ui/                    # HUD и инвентарь
  /server                    # Fastify + Socket.IO
    src/
      ecs/                   # компоненты и мир
      sim/                   # авторитарная симуляция, боты
      net/                   # обвязка сокетов
      persistence/           # заглушки хранилищ
  /shared                    # общие типы и протокол
```

## Управление и UX
- **Левая половина экрана** — виртуальный джойстик передвижения.
- **Правая половина** — свайп камеры и кнопки действий:
  - «Прыжок» — прыжок
  - «Удар» — ближняя атака
  - «Взаим.» — сбор/взаимодействие
  - «Инвентарь» — открыть/закрыть панель (отображается всегда для М1)
- **Клавиатура (desktop dev):** WASD — движение, Space — прыжок, F — удар, E — взаимодействие, I — инвентарь.

UI адаптируется под DPR мобильных экранов, использует HTML/CSS поверх WebGL-канваса.

## Игровые системы М1
- Авторитарный сервер с тикрейтом 20 Гц, снапшоты 10 Гц. Клиент интерполирует позиции игроков и сущностей.
- Карта 500×500 м, процедурный рельеф дюн, расставленные скалы и башни.
- Параметры игрока: HP, Hunger, Thirst, Stamina, температура окружающей среды. Температура влияет на расход жажды и стамины.
- Сбор ресурсов (камень, дерево, металл, вода, волокно), респавн нод каждые 120 сек.
- Крафт сервер-сайд, рецепты из таблицы ниже, инвентарь 4×5 слотов, стаки по типам.
- Моб «пустынная крыса» патрулирует и агрится в радиусе 18 м, наносит урон при контакте.
- Смерть сбрасывает часть слотов, респавн через 10 сек на случайной точке пустоши.
- День/ночь — синусоидальная кривая освещения, температура днём выше.
- PWA: manifest + service worker (кеш статических ассетов, офлайн-страница).
- Логирование FPS и ключевых метрик в консоли (devtools), сервер логирует тик/температуру каждые 5 сек.

## Сетевой протокол
### Client → Server (`input`)
```json
{
  "op": "input",
  "at": 1695642000.12,
  "seq": 152,
  "move": { "0": 0.7, "1": 0.0 },
  "look": { "0": -3, "1": 1 },
  "actions": { "jump": false, "hit": true, "interact": false, "inventory": false }
}
```
### Server → Client (`state`)
```json
{
  "op": "state",
  "tick": 4201,
  "you": "p_abc",
  "players": [
    { "id": "p_abc", "pos": [10.2,0,-3.1], "rotY": 91, "hp": 74 },
    { "id": "p_xyz", "pos": [15,0,-2], "rotY": 270, "hp": 100 }
  ],
  "entities": [
    { "id": "n_12", "t": "rock", "pos": [20,0,-5] },
    { "id": "ai_3", "t": "rat", "pos": [9,0,-1], "hp": 30 }
  ],
  "timeOfDay": 0.54,
  "temperature": 33.2
}
```

Фактическая реализация использует типизированные интерфейсы из `shared/types.ts`, снапшоты интерполируются клиентом. Все проверки крафта, урона и сборов происходят на сервере.

## PWA и оптимизации
- Service Worker кеширует shell и офлайн-страницу, лениво загружаются игровые чанки.
- Vite сборка с `splitVendorChunkPlugin`, ручное разбиение на чанки `babylon` и `net`.
- Lazy-load сцены (`import("./scenes/gameScene")`) и отложенная регистрация SW.
- Бюджет draw calls ≤ 200 достигается батчингом примитивов и упрощёнными материалами.

## Перспектива М2
- Подключение Prisma + SQLite для хранения инвентаря и построек.
- Переключатель настроек графики, дополнительные погодные зоны и пост-эффекты.
- Расширенная физика с помощью cannon-es: slope limit, step offset.
